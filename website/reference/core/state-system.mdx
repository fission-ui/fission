# State system

Fission uses a single source-of-truth pattern:

- `AppState`: data for the current UI + product model,
- `Action`: typed event intents,
- `Reducer`: deterministic state mutation,
- `ReducerContext`: runtime metadata and async completion hooks.

## Flow in one sentence

`Action -> reducer(context, state) -> new state -> build() -> UI`

## Why teams use this

- fewer hidden dependencies,
- reproducible behavior for tests,
- easy migration from one screen to another because every screen uses the same model.

## Guardrails

Reducers should:

- stay pure,
- avoid I/O,
- stay local to one domain concept.

If you need async, move that work behind [commands, services, and jobs](/reference/core/commands-services-jobs).

# Commands, services, and jobs

This reference page is the contract sheet for async behavior.

- **Job**: one-shot execution tied to a request/response pair.
- **Service**: long-lived process with start/stop and events.
- **Command**: explicit instruction to an active service.

## Data model

Use completion actions to bring async results back through normal reducer flow:

```text
action -> async trigger -> completion action -> reducer -> render
```

## Registering async behavior

All async registration is done on app startup through `with_async` so runtime wiring is centralized.

## When each primitive is appropriate

- choose a **job** when the work is isolated and returns once,
- choose a **service** when work is ongoing or stateful,
- choose a **command** for explicit control signals.

## Test points to use

- reducer unit tests,
- completion payload decode via `ActionInput`,
- behavior tests through `TestHarness` and `LiveTestClient`.

# Start here

If this is your first day with Fission, your goal is simple: get one small interaction working and learn the same pattern once.

Here is the shortest path that gets you to a meaningful app loop quickly:

1. Install the CLI and create a baseline app.
2. Make a small state change and confirm it works.
3. Add one second interaction.
4. Decide where async work belongs.
5. Generate your next target only when behavior is stable.

## How Fission targets your workflow

Fission is designed around this rule: *build trust in state first, then scale runtime coverage*.

Use one path now and come back later to switch target mode once behavior is stable.

<Tabs>
<TabItem value="default" label="Default path" default>

Start with a runnable app generated from the current machine and Rust toolchain:

```bash
cargo install fission-cli
fission init my-fission-app
cd my-fission-app
cargo run
```

This validates your app loop before any platform-specific generated host is added.

</TabItem>
<TabItem value="custom" label="Local checkout path">

Use this for local crate development and deeper framework diagnostics:

```bash
git clone https://github.com/worka-ai/fission.git
cd fission
FISSION_CHECKOUT="$PWD"
cargo install --locked --path "$FISSION_CHECKOUT/crates/tools/fission-cli"
cd ..
fission init my-fission-app --local-path "$FISSION_CHECKOUT"
cd my-fission-app
cargo run
```
</TabItem>
</Tabs>

## Add targets when behavior is ready

You can add platforms as needed with the same project manifest (`fission.toml`):

```bash
cargo fission add-target web ios android
```

Generated targets are run from the generated app root:

- `./platforms/web/run-browser.sh`
- `./platforms/ios/run-sim.sh`
- `./platforms/android/run-emulator.sh`

Read each generated target README before testing each runtime.

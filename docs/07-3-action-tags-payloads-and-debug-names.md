# 7.3 Action Tags, Payloads, and Debug Names

This section defines how actions are identified, parameterized, and inspected at runtime.
Action tags, payloads, and debug names together provide a stable, type-safe, and tool-friendly representation of user intent.

These concepts exist to avoid stringly-typed actions while remaining observable and debuggable.

---

## 7.3.1 Action Tags

### Definition

An **Action Tag** is the canonical, stable identifier for an action.

Properties:
- globally unique within the application,
- derived deterministically,
- opaque at runtime,
- comparable and serializable.

Tags are the primary identity used for routing and validation.

---

## 7.3.2 Tag Generation

Action tags are generated by `#[fission_action]`.

Derivation inputs include:
- crate identity,
- action type name,
- optional explicit version.

This ensures:
- no accidental collisions,
- stability across builds and platforms,
- reproducibility in snapshots and replay.

Tags are never authored manually as strings.

---

## 7.3.3 Built-in vs User-Defined Tags

There are two classes of tags:

- **Built-in Tags**  
  Provided by the framework (e.g. Activate, Increment).

- **User-Defined Tags**  
  Defined via `#[fission_action]`.

Both are treated identically by the Core runtime.

---

## 7.3.4 Action Payloads

### Definition

An **Action Payload** is structured data associated with an action invocation.

Payloads:
- are optional,
- are strongly typed,
- are validated against a schema,
- are immutable during dispatch.

Payloads carry *parameters*, not behavior.

---

## 7.3.5 Payload Schemas

Payload schemas are derived from action types.

Rules:
- field names and types define the schema,
- optional fields must have defaults,
- schemas are versioned implicitly with the action.

Schema mismatches are deterministic dispatch errors.

---

## 7.3.6 Payload Serialization

Payloads are fully serializable.

This enables:
- action logging,
- deterministic replay,
- cross-process testing,
- snapshot comparison.

Serialization format is stable and versioned.

---

## 7.3.7 Debug Names

### Definition

A **Debug Name** is a human-readable identifier for an action.

Examples:
- "Activate"
- "Increment"
- "SetCount(value=3)"

Debug names are:
- derived automatically,
- stable but non-semantic,
- used only for diagnostics and tooling.

---

## 7.3.8 Debug Name Generation

Debug names are generated from:

- the action type name,
- payload field names and values (when present).

Rules:
- generation is deterministic,
- formatting is standardized,
- localization is not applied.

Debug names are never used for routing or equality.

---

## 7.3.9 Instrumentation and Tracing

Action tags and debug names enable:

- structured action logs,
- readable traces,
- correlation with reducers and state changes.

Instrumentation may record:
- tag,
- debug name,
- payload,
- target node identity.

---

## 7.3.10 Testing Implications

Tests may refer to actions by:

- type (preferred),
- tag (for low-level assertions),
- debug name (for diagnostics only).

Tests must not rely on debug name formatting for correctness.

---

## 7.3.11 Validation Rules

The runtime validates that:

- action tags are known,
- payloads match schemas,
- debug metadata is well-formed.

Invalid actions are rejected deterministically.

---

## 7.3.12 Summary

Action tags, payloads, and debug names together provide:

- strong, stable identity for actions,
- structured parameter passing,
- rich observability for debugging and tooling,
- zero reliance on fragile string identifiers.

They complete the action representation layer.
